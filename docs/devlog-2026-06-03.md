## 1. Blok-rygrad: Fase A-F

Brugerens fremstilling om morgenen: *"Vi har et blok-system, men det virker ikke. Lad os tænke det helt igennem — det er rygraden i systemet."*

Den oprindelige plan: vinder af en blok skal gemmes, leaderboard skal vise antal sejre, dashboard skal vise "næste blok lukker om X etaper", og når man opretter et nyt spil, skal man kunne se hvilke blokke der bliver lavet. Inden vi gik i kode, lagde vi 6 faser ud:

| Fase | Hvad |
|---|---|
| A | Schema — `stage_number_min/max`, `winner_user_id`, `winner_points`, `finalized_at` på cycling_blocks |
| B | Generator — `generateCyclingBlocks` læser `rest_days` og populerer felter ved oprettelse |
| C | Lifecycle — fuld `upcoming → active → finished` automatik |
| D | Persistens — auto-finalize + snapshot til ny `cycling_block_results`-tabel |
| E | UI — sejre på profil + dashboard "næste blok lukker om X etaper" |
| F | Wizard — preview af blokke før oprettelse |

### Fase A+B + leaderboard læser fra persistens (PR #239)

Migration `20260528_cycling_blocks_metadata.sql` tilføjer kolonnerne og opretter `cycling_block_results` (immutable standings-snapshot). Indeks på `(parent_block_id, status)` for hurtige aktiv-sub-blok-queries.

`computeSubBlockRanges` (ny pure function i [lib/cyclingBlocks.ts](../lib/cyclingBlocks.ts)) regner sub-blok-grænser ud fra `cycling_races.rest_days`:

```ts
function computeSubBlockRanges(stages, restDays) {
  // For hver hviledag: sidste etape før hviledagen lukker en uge,
  // første etape efter starter den næste.
  // Fald tilbage til "3 lige uger" hvis rest_days mangler.
}
```

[scripts/backfill-cycling-blocks.ts](../scripts/backfill-cycling-blocks.ts) populerer eksisterende blokke + finaliserer færdige sub-blokke + snapshotter standings. Idempotent (UPSERT) med `--dry-run`, `--game <id>` og `--regenerate-subblocks` flags.

Vigtigste skift i samme PR: `buildCyclingLeaderboard` læser nu `winner_user_id` direkte fra cycling_blocks (top + sub) i stedet for at recomputere scores live. Top- og sub-blok-sejre tæller hver for sig — Giro Uge 1 og Uge 2 giver hver én sejr til vinderen. Fjernede 40 linjer død live-aggregation (`countHistoricalCyclingBlockWins` + raceToBlock-map).

Backfill mod prod afslørede 2 vundne uger: vinderen af Giro Uge 1 (1029,99 pt) og Uge 2 (627,75 pt) — begge Nikolaj — fik B. sejr: 2 i leaderboard'et.

### Fase C+D + UI-label (PR #242)

`updateCyclingBlockStatuses` omskrevet til fuld 3-status-lifecycle:

```
upcoming → active:   første relevante etape har resultater
active → finished:   alle relevante etaper har resultater
```

Auto-finalize ved overgang til finished:
1. `computeBlockStandings()` summer total_points for hver bruger
2. `snapshotBlockResults()` UPSERTer standings til `cycling_block_results`
3. `winner_user_id` / `winner_points` / `finalized_at` sættes på blokken

Pure helper-fil [lib/cyclingBlockStandings.ts](../lib/cyclingBlockStandings.ts) — genbrugbar i scripts og senere wizard.

`CyclingBlockStanding` viser nu en gylden pille på top-1:
- `active blok` → **"Fører"** (live spænding)
- `finished blok` → **"Vinder"**

### Fase E — profil-sejre + dashboard (PR #243)

[components/cycling/CyclingProfileWins.tsx](../components/cycling/CyclingProfileWins.tsx) — server-component der læser `cycling_blocks.winner_user_id` direkte. Pr. sejr: blok-navn, spil-navn, dato, point + trofæ-ikon. Inkluderer både top-blok-sejre og sub-blok-sejre.

Dashboard "næste blok lukker" — `activeBlockByGame` udvidet til cykel-spil:
- Find aktiv (sub-)blok
- Tæl resterende ikke-uploadede etaper
- Vis "Giro Uge 3 · 2 etaper tilbage"

Foretrækker sub-blok over top-blok når en sub-blok er aktiv.

### Fase F — wizard blok-preview (PR #246)

Backend `app/api/cycling/preview-blocks/route.ts` — POST endpoint der tager race_ids og returnerer planlagt struktur. **Genbruger `computeSubBlockRanges`** så preview = generator-resultat 1:1.

UI `CyclingBlocksPreview.tsx` — client component der fetcher ved selection-ændringer, viser pr. blok: navn, etape-antal, GT-badge, sub-blokke med stage-range. Gylden advarsels-callout hvis Grand Tour mangler hviledage.

Indsat i NewCyclingGameForm efter sæsonkalender + AddRacesForm før knap-bar.

## 2. Reglebogen omskrevet 2 gange

### v1 — rolle-første struktur (PR #241, første del)

Den eksisterende guide-side var 8 sektioner med 366 linjer. Reglerne fremstod fragmenterede — kategori, profil, won-how, jersey, GC og spurt-tog forklaredes hver for sig. Brugerne skulle selv samle puslespillet og rapporterede *"der er mange regler"*.

Brugerens valg af struktur: **rolle-første guide** med konkrete eksempler pr. rolle + inline-tooltips i selve spillet.

Centraliserede konstanter til [lib/cyclingScoringConstants.ts](../lib/cyclingScoringConstants.ts) — én kilde til sandhed for scoring og reglebog:

```ts
export const POSITION_POINTS, CAT_MULTIPLIER, GC_MULTIPLIER, JERSEY_POINTS
export const TEAM_BONUS_DEFAULT, EQUIPIER_TEAM_BONUS, DOMESTIQUE_BONUS
export const TRAIN_BONUS_PER_LEADOUT, TRAIN_MAX_LEADOUTS
export function getBasePoints, getGcMultiplier, getSprinterMultiplier, ...
```

Hero med grundformlen øverst: `Basispoint × Kategori × Profil × Spurt-tog + bonus − straf`. Klikbare rolle-cards der scroll-anchorer til pr. rolle. Pr. rolle: tagline, formel, multiplier-tabeller, eksempler med beregning + resultat (gylden highlight på de vigtige), bonusser, straffe, strategi.

Score-breakdown i tooltip — `cycling_scores` fik nye kolonner `cat_multiplier`, `profile_multiplier`, `train_multiplier`. Tooltip viser nu hver komponent som egen linje når aktiv. Spurt-tog højlightes guld. Det forklarede pludselig hvorfor Stæhrs Milan-score på E21 (145,4 pt) var højere end Nikolajs (122,0): Stæhr havde Ciccone (Lidl-Trek) som equipier → spurt-tog ×1.2.

Inline-links fra spillet: gul "?"-pille i LineupBuilder rolle-modal → guide-anchor. "Sådan beregnes point →" link i score-tooltip → guide-anchor.

### v2 — forenklet sprog uden rytter-navne (PR #244)

Brugerens feedback efter v1: *"Folk synes der er mange regler... men i virkeligheden handler det om at vi er dårlige til at forklare og vise."* Plus: *"Du behøver ikke bruge ryttere for at lave eksempler."*

Reglerne blev IKKE ændret — kun forklaringerne:

| Før | Efter |
|---|---|
| "Den rytter du satser på vinder eller kommer højt op. Klassisk: en topstjerne i Kategori 1 (Pogačar, Vingegaard, van der Poel)..." | "Den rytter du tror vinder. Får point efter placering ganget med kategori." |
| "Stabil men dyr på kat 1. Kat 3 leader giver markant højere udbytte hvis de leverer..." | "Højere kategori = højere risiko, højere udbytte." |
| "Kat 2 Milan vinder flat, 1 leadout" | "Kat 2 sprinter vinder flad — med 1 équipier fra samme hold" |

Tagline kortet til 2-4 ord. Beskrivelse til 1-2 sætninger. Eksempler bruger abstrakte ryttere. Strategi til 1 sætning.

Småfix bagefter:
- Float-rounding: `pt()` helper der altid runder til 1 decimal. Tidligere viste 50 × 1.7 × 2.8 sig som `237.99999999999997 pt` pga. float-precision.
- *"Spillets spurter"* → *"Spillets sprinter"* — konsekvent dansk cykel-term. "Spurt-tog" og "Massespurt" bevaret (de er navngivne mekanikker).
- Joker-strategi kønsneutral: "gør hende til Joker" → "godt sted til en risikabel rytter du tror på".

## 3. Ingen minus-point: én beslutning, mange konsekvenser

### v1: fjern fra reglebog (PR #245, del 1)

Brugerens feedback ved sidste runde af reglebog-fixes: *"Ingen DNF det har vi vidst heller ikke i reglerne."* Først troede jeg han mente *"vi viser det forkert i UI'et"* — men nej. Han meddelte at vi ikke skulle have minus-point eller straffe overhovedet.

Konsekvenser:
- Alle `penalties: [...]` i ROLE-data slettet
- "DNF-straf" SubCard fjernet fra Mekanikkerne-sektionen
- Joker-beskrivelse trimmet: ikke længere "Tæller ALDRIG minus", bare "Wildcard"
- Joker DNF-eksempel erstattet med "uden for top-20"

### v2: fjern fra scoring-koden (PR #245, del 2)

Det var ikke nok at fjerne fra reglebogen — koden i [lib/calculateCyclingPoints.ts](../lib/calculateCyclingPoints.ts) havde stadig aktiv DNF-straf-logik:

```ts
if (isDnf && !isJoker) {
  const wouldScore = rolePoints + jerseyPts + teamBonus
  dnfPen = wouldScore > 0 ? -Math.round(wouldScore * DNF_PENALTY_PCT * 10) / 10 : DNF_PENALTY_MIN
}
```

Fjernet. `dnf_penalty` sættes altid til `0` ved upsert. `DNF_PENALTY_PCT`/`DNF_PENALTY_MIN` konstanter fjernet. Re-scored alle 21 Giro-etaper for at nulstille eksisterende DNF-straffe i historiske scores.

Bevaret: **"Leader udgår"** trigger for Lieutenant (×2.8) — det er en BONUS, ikke en straf. `abandon_type`/`dnf`-feltet i `cycling_results` bevares så vi stadig tracker udgang, vi bare straffer det ikke.

### v3: ryd dead kode (PR #247)

`bench_penalty` + `dnf_penalty` kolonner var nu altid 0, men selekteres, returneres i API'et og indgik i `total_points`-formlen. Migration `20260603_cycling_scores_drop_penalty_columns.sql` dropper kolonnerne og genskaber `total_points` GENERATED column:

```sql
ALTER TABLE cycling_scores DROP COLUMN IF EXISTS total_points;
ALTER TABLE cycling_scores DROP COLUMN IF EXISTS bench_penalty;
ALTER TABLE cycling_scores DROP COLUMN IF EXISTS dnf_penalty;
ALTER TABLE cycling_scores ADD COLUMN total_points numeric GENERATED ALWAYS AS (
  (base_points * role_multiplier * gc_multiplier)
  + role_bonus + jersey_points + team_bonus
) STORED;
```

Fjernet fra `lib/calculateCyclingPoints.ts` (ScoreRow + upsert + 60+ linjer SQL-historik fra header), `app/api/games/[id]/cycling/lineup/route.ts` (SELECT + response), `components/cycling/LineupResults.tsx` (Score type, tooltip-linjer, hele `BenchTooltip` der altid kun viste tomt indhold), `components/cycling/LineupBuilder.tsx` (ScoreEntry type).

### v4: ret stale ROLE_TOOLTIPS (PR #249)

Endnu en efterklang. ROLE_TOOLTIPS på rolle-pille hover sagde stadig *"DNF: -50% af score"* og *"Immun mod DNF-straf"* — direkte modsigelse af reglerne efter PR #245. Opdateret til at matche reglebogen + bruge sprinter/bjerg/etc. terminologi.

Lærdom: én beslutning kan trække 4 PR'er hvis konceptet er spredt over schema, kode, reglebog og tooltips. Vigtigst at fange dem ALLE samme dag mens beslutningen er frisk — ellers bliver stale-teksterne hængende.

## 4. Dauphiné-fix: kun ægte Grand Tours får sub-blokke

Sent på dagen testede brugeren wizard-preview ved at tilføje Critérium du Dauphiné til sit gameroom. Preview viste:

```
Critérium du Dauphiné
8 etaper · 3 sub-blokke

Uge 1   Etape 1-3 · 3 etaper
Uge 2   Etape 4-6 · 3 etaper
Uge 3   Etape 7-8 · 2 etaper
```

Brugerens kommentar: *"Dauphiné er kun 8 etaper over en uge. Vi har kun de tre store GT's der er over 3 uger, som skal skæres i flere blocks."*

Roden: `computeSubBlockRanges` faldt tilbage til "3 lige uger" når `restDays` manglede OG `totalStages >= 4`. Det rammede Dauphiné, Paris-Nice, Tour de Suisse — alle uger-lange stage races der har INGEN naturlig ugentlig opdeling.

Fix (PR #248): ny eksport `GRAND_TOUR_MIN_STAGES = 14`. `computeSubBlockRanges` returnerer `[]` under denne tærskel.

| Stage race | Etaper | Før | Efter |
|---|---|---|---|
| Giro/Tour/Vuelta | 21 | 3 sub-blokke | 3 sub-blokke ✅ |
| Dauphiné | 8 | 3 sub-blokke ❌ | Ingen sub-blokke ✅ |
| Paris-Nice | 8 | 3 sub-blokke ❌ | Ingen sub-blokke ✅ |
| Tour de Suisse | 8 | 3 sub-blokke ❌ | Ingen sub-blokke ✅ |

Preview-API'et importerer samme konstant — én kilde til threshold.

Lærdom: hvad jeg troede var en pragmatisk fallback ("3 lige uger som fallback") var i virkeligheden en regel der gjaldt for hele 80% af aktuelle stage races. Konsekvensen er først synlig når en bruger faktisk tilføjer et af de berørte løb.

## 5. Lukket blok skal forsvinde fra gameroom

Brugerens næste rapport: *"Giro er lukket, Nikolaj er erklæret som vinder. Men klassementet og de ting står stadig i Gameroom."*

Diagnostik i [app/games/[id]/page.tsx](../app/games/[id]/page.tsx):

```ts
// Find brugerens aktive blok (fra første squad med cycling_block_id)
const cyclingBlockId = userSquads?.map((sq) => sq.cycling_block_id).find(Boolean) ?? null
```

`cyclingActiveBlock` blev sat ud fra brugerens **squad's** `cycling_block_id`, ikke spil-status. Stæhrs squad var i Giro-blokken. Når Giro var status='finished', viste klassement, lineup-default og standings STADIG Giro.

Fix (PR #250): efter `cyclingBlocks`-fetch, override `cyclingActiveBlock` hvis den er finished:

```ts
const currentInList = cyclingBlocks.find((b) => b.id === cyclingActiveBlock!.id)
if (currentInList?.status === 'finished') {
  const nextActive = cyclingBlocks.find((b) =>
    b.parent_block_id === null && b.status !== 'finished'
  )
  if (nextActive) cyclingActiveBlock = nextActive
}
```

Plus: `classements` filtreres til kun den aktive blok via `lineupRaces.cycling_block_id`. Tidligere viste `CyclingGameroom` alle klassementer i `lineupClassements` på tværs af alle blokke — nu kun den aktive.

Effekt: Giro-klassementet (rosa trøje, pointtrøje, etc.) forsvinder nu Giro er finished. Lineup-builder default'er til Tour eller Dauphiné. Standings følger.

## 6. Sæson-overblik

Samme brugerrapport, anden del: *"Vi mangler ligesom et overordnet leaderboard, hvor vi kan se stillingen i gameroom på sæson basis — antallet af samlet point vundet, antallet af blokke vundet og antallet af samlet sejr."*

Den eksisterende `Leaderboard.tsx` viste:
- `R. sejr` (round wins = stage wins i nuværende blok)
- `R. point` (round points = current block points)
- `B. sejr` (block wins — alle blokke)
- `B. point` (block points = nuværende blok kun)

Mangler: **sammenlagt point på tværs af ALLE blokke**. Brugeren valgte:
- Alle blok-vindere tæller ens i én "Sejre"-kolonne
- Tilføj som ekstra blok (behold nuværende Leaderboard)

Ny komponent [components/cycling/CyclingSeasonOverview.tsx](../components/cycling/CyclingSeasonOverview.tsx):

```
SÆSON-OVERBLIK
   Spiller         Sejre    Sammenlagt
1  Nikolaj           2       1657.7 pt
2  Stæhr             —        892.4 pt
```

Backend: `LeaderboardEntry` får ny `total_points: number`. `buildCyclingLeaderboard` beregner `totalPointsByUser` i samme score-loop som de eksisterende felter — kun **uden** current-block-filter. `buildEntries` tager nu optional `totalPointsByUser` parameter. Fodbold falder tilbage til `block_points` som approksimation indtil vi udvider fodbold-buildere tilsvarende.
