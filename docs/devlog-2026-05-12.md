## 1. Stripe-medlemskab + paywall

### Subscription-stack live (commit `2a3a311`)
Hele subscription-flowet på plads og deployet til prod:

**Pris**: €1/måned recurring, Stripe Live mode med eget product + price.

**DB-migration** ([20260511_stripe_subscriptions.sql](../supabase/migrations/20260511_stripe_subscriptions.sql)):
- `profiles.subscription_status` enum (`none / active / comped / past_due / canceled`)
- `profiles.stripe_customer_id`, `stripe_subscription_id`, `subscription_current_period_end`

**Backend**:
- [lib/stripe.ts](../lib/stripe.ts) — SDK-init med API-version `2026-04-22.dahlia` + Stripe→intern status-mapper
- `POST /api/stripe/checkout` — opretter Checkout Session, persisterer `stripe_customer_id` ved første session
- `POST /api/stripe/portal` — Customer Portal redirect
- `POST /api/stripe/webhook` — håndterer `checkout.session.completed`, `customer.subscription.{created,updated,deleted}`, `invoice.payment_failed` med signature-verifikation. Handlers er idempotente by-design (alle sætter status til target-state, ikke increment) → Stripe-retries giver wasted DB writes men ingen korruption
- `PATCH /api/admin/users/[id]/comp` — admin-toggle for gratis adgang (audit-logged)

**UI**:
- [/subscribe](../app/subscribe/page.tsx) — editorial pris-side med bodega-receipt + checkout-knap
- [/profile/billing](../app/profile/billing/page.tsx) — viser status + Customer Portal-knap
- Admin Users-tab: medlemskab-kolonne + "Gratis adgang"-toggle pr. bruger

**Paywall**: middleware.ts redirect'er til `/subscribe` hvis status ∉ (`active`, `comped`). Allow-list: `/`, `/login`, `/register`, `/logout`, `/subscribe`, `/profile/*`, `/api/stripe/*`, legal-sider. Admins bypasser paywall.

### Bug-fixes efter live-launch (commits `4686add`, `614b391`)

3 huller fanget af post-launch audit:

1. **Stripe Portal `return_url` pegede på `/account`** (findes ikke) → 404 efter portal-session. Fixet til `/profile/billing`.
2. **Middleware allow-listede engelske legal-routes** (`/privacy`, `/terms`, `/cookies`, `/about`, `/contact`) som ikke eksisterer. De faktiske routes er danske (`/vilkaar`, `/privatlivspolitik`, `/cookie-politik`, `/om`, `/faq`) — opdateret. Auth-flows (`/forgot-password`, `/reset-password`, `/verify-email`) også tilføjet.
3. **`past_due`-status kunne starte ny Checkout** → risikerede dobbelt-charge når original sub recovery'er. Bloker nu med besked om at fixe kortet via Customer Portal.

### Lærdom: vendor-onboarding er nemmere end forventet
- Stripe Sandbox vs Live separation kostede én test-betaling før vi havde det rigtige price ID (sandboxes har egne objekt-ID'er — kan ikke krydses)
- Webhook-event-lytning kræver `stripe listen --forward-to localhost:3000/...` i en separat terminal lokalt. Webhook-secret er per-listen-session
- Customer Portal-config i Stripe-dashboard skal aktiveres separat før portal-routen virker

## 2. Mobile-first admin

### Struktur-refaktor: 3-sektion nav (commit `cd5c07f`)
Admin-modulet var organiseret som 2 sportsgrene (Fodbold + Cykling) med 9+4 tabs. Brugere (cross-cutting), Audit og Logs lå ulogisk under Fodbold. Refaktoreret til:

| Section | Tabs |
|---|---|
| **System** ⚙ | Brugere, Audit log, Logs |
| **Fodbold** ⚽ | Overblik, Kampprogrammer, Sæsoner, Spilrum, Mesterskabet, LIVE TEST |
| **Cykling** 🚴 | Oversigt, Ryttere, Løb, Spilrum |

Default-tab er `System → Brugere` — matcher den primære mobile use-case (comp/suspend fra telefonen). URL-param skiftet `?sport=…` → `?section=…` med bagudkompat på gamle bookmarks.

### Tab-by-tab mobile-konvertering (commits `0075eea`, `94d5b96`, `9418b09`, `876f6c9`, `3c49783`)

Alle 11 tabs har nu dedikeret mobile-layout via `hidden md:block`-mønstret — desktop-layout uændret.

**Tab-nav**: horizontal scroll på mobil med `whitespace-nowrap shrink-0` + skjult scrollbar.

**Brugere** ([AdminUsersTab](../components/admin/tabs/AdminUsersTab.tsx)): card-list med navn/email/medlemskab-badge/suspended-badge øverst + 2 store touch-knapper (`Gratis adgang | Suspendér`) i grid-cols-2 layout.

**Audit log + Logs**: tighter horizontal padding (`px-3 sm:px-5`), filter-rækken wrapper, py-2 touch-targets på filter-knapper.

**Cykling Oversigt** ([AdminCyclingDashboardTab](../components/admin/tabs/cycling/AdminCyclingDashboardTab.tsx)): 3 sync-knapper stacked fuld-bredde på mobil med `py-3` touch-targets. Response-besked vises **under hver knap** i en flex-col container — før blev alle 3 messages dumpet i en wraplet linje til højre.

**Cykling Løb** ([AdminCyclingRacesTab](../components/admin/tabs/cycling/AdminCyclingRacesTab.tsx)): race-cards med status-dropdown + startlist-knap + expand-stages toggle. Stages inde i expanded card vises som kompakte stage-cards i stedet for 9-kolonne sub-table. Startlist-modal slider op fra bunden på mobil (`items-end sm:items-center`).

**Cykling Ryttere** ([AdminCyclingRidersTab](../components/admin/tabs/cycling/AdminCyclingRidersTab.tsx)): rider-cards med foto + navn + kat-badge + manglende-logo-indikator. Filter-rækken stacker vertikalt, kat-filter + foto-filter side om side.

**Fodbold Overblik** ([AdminOverviewTab](../components/admin/tabs/AdminOverviewTab.tsx)): cron-knapper i 2-col grid med py-3 touch-targets. MatchList-tabel konverteret til card-layout.

**Fodbold Spilrum** ([AdminGamesTab](../components/admin/tabs/AdminGamesTab.tsx)): game-cards med slet-flow inline (confirm-state bruger grid-cols-2 i stedet for inline-row).

**Fodbold Sæsoner** ([AdminSeasonsTab](../components/admin/tabs/AdminSeasonsTab.tsx)): tournament-cards + season-cards med rediger-knap fuld-bredde.

**Fodbold Mesterskabet** ([AdminChampionshipTab](../components/admin/tabs/AdminChampionshipTab.tsx)): ombytter kolonne-rækkefølge på mobile (rounds vises før kamp-browser) via `order-1 / order-2` så brugeren ser runder først (action-pålægger).

**Fodbold LIVE TEST** ([LiveTestTab](../components/admin/tabs/LiveTestTab.tsx)): live-match card-list med score som hovedfokus + bold match-id som footer-detalje.

**Fodbold Kampprogrammer** ([LeagueHubClient](../components/admin/LeagueHubClient.tsx)): stats-banner i 3-col grid, Sync alle-knap fuld-bredde. Sæson-tabel → card-list med Sync-knap + feedback inline.

## 3. Landing-migration + brand-narrative shift

Progressiv migration af /landing-v2-sektioner til produktion (`/`). Hver sektion extract'et til shared `components/landing/*.tsx` så begge sider bruger samme kilde.

### Brand-narrative shift: gamemode-first (commit `a8d7314`)

Den **største ændring** i landing-narrativet: vi præsenterer nu produktet som **gamemodes**, ikke discipliner. Det forbereder også for kommende discipliner (F1, Tennis).

| Før | Efter |
|---|---|
| Disciplin = Cykling, Fodbold | Gamemode = Fantasy, Konsensus-betting |
| Title: "Cycling Manager", "Fodbold" | Title: "Fantasy", "Konsensus-betting" |
| 2 discipliner | 2 formater, hver med flere discipliner |

Når F1 launcher: bevarer **Fantasy** som hovedtitel, opdaterer eyebrow til "Cykling + F1". Når tennis launcher: bevarer **Konsensus-betting**, opdaterer eyebrow til "Fodbold + Tennis". Konsistensen i brandfortælling holdes uanset hvor mange discipliner der kommer.

1. **Port direkte fra /landing-v2** — venstre kolonne "Du betaler til oplevelsen, ikke til udvikleren" + højre kolonne med 4× "INKLUDERET"

## 4. Kendte issues / til senere

- **Cookie-banner mangler** — vi har `/cookie-politik` men ingen consent-banner. EU-compliance dictates one, especially since Sentry + Resend bruger cookies/tracking. P3 men relevant.
- **HowItWorksSection er code-død** men findes stadig i `components/landing/` — beholder filen i tilfælde vi vil tilføje en step-by-step nedenfor senere.
- **ProductsSection skalering med 3+ discipliner**: Den nuværende design antager 2 gamemodes. Når Fantasy også har F1 og Tipping har Tennis, skal vi vælge: behold 2 kort (én pr. gamemode) med multi-disciplin eyebrow, ELLER ekspandér til 4 kort (én pr. disciplin). 2-kort skalerer bedst visuelt.
- **Stripe live mode**: kun €1 test-charge gennemført i smoke-test (refundered). Ingen rigtige kunder endnu — verifikation kommer naturligt når vennegrupperne abonnerer.
- **`is_admin` query fejler hvis profiles-tabellen er nede**: middleware fall-back gør `profile` null → admins får 404 på /admin. Skulle ideelt være forsøg-retry eller failsafe-mode. Lavt sandsynligt scenarie.
